01What Tales is

Tales is a lightweight Rust orchestrator that runs AI coding CLIs — Claude Code, Codex, and Open Code — as subprocesses so they collaborate on the same project. They debate a plan in a live terminal, each emits a recommendation for who should execute it, and no editing or execution hand-off starts until you confirm.

It's not a model or an API wrapper. It drives the CLIs you already have installed and authenticated, normalizes their very different I/O into one event stream, and puts a human at the decision gate.

Two strong models beat one

A single model commits to its first framing and rarely escapes it. Tales puts two models with different priors in the same room: one drafts, the other critiques across alternating turns, and the disagreement surfaces assumptions a solo run buries.

The analogy to OpenRouter's model fusion and AWS Kiro is narrow and deliberate: combine the strengths of more than one model instead of betting everything on one — fusion blends answers across models, Kiro splits spec-then-build across agents. Tales applies the idea to planning (the live drafter/critic discussion) and gives you the tools to apply it to execution too (isolated git worktrees), with a human on the gate.

It's deliberately model- and tool-agnostic: any AI coding CLI can join behind one small adapter (Claude Code, Codex, Open Code today), and the room doesn't have to be uniform. You can tier it — strong, expensive models do the planning, a cheaper/faster model implements the agreed plan (see --execute-model). Spend on judgment, not on typing.

What you get:

  • A drafter/critic discussion you can read as it streams and interject into.
  • A confidence-weighted recommendation of who should execute — advisory, never automatic.
  • Git-worktree isolation for the executor, so a result is always a reviewable diff.

What it is not: an autonomous agent that ships code on its own. The recommendation is a suggestion; the gate is yours.

02Install & prerequisites

  • Rust toolchain (stable) to build the workspace.
  • Developed and tested on macOS and Linux. tales open (the new-window launcher) is macOS-only; the other commands are Unix-oriented.
  • At least one agent CLI on your PATH, installed and authenticated. The default discussion pairs two, so for a single tool you pass explicit flags (see below).

03The interactive terminal

Type tales to open a multi-pane terminal workspace: a Tales orchestrator pane plus sibling shell and agent-CLI panes.

  • Plan. In the Tales pane, type your task and press Enter; the drafter and critic discuss and draft a plan in place, streamed token-by-token.
  • Open panes as needed. Ctrl-N shell · Ctrl-X Codex · Ctrl-L Claude Code · Ctrl-O Open Code. Tab switches focus.
  • Observe / intervene. A focused agent pane receives your keystrokes directly. Ctrl-A sends an explicit approval, but only when that pane is waiting on one.
  • Confirm & hand off. After the discussion, agents emit recommendations and Tales surfaces the aggregated pick. At the gate, Enter or /confirm <n> launches the executor in its own live CLI pane and sends it the plan; or focus an existing agent pane and press Ctrl-S to send the plan there.
Two execution paths. In the workspace, confirming hands the plan to a live, interactive CLI pane you watch and steer. The scriptable tales run and the classic planner run the executor headlessly through the core. Same plan, different driver.
Heads up — live panes share a directory. Worktree isolation is automatic only for tales run --worktree. Live executor panes all start in the same working dir, so if you open several to run in parallel, isolate them yourself (separate worktrees/branches) to avoid clobbering.

Prefer one focused view? tales-tui --classic is the original connect → prompt → plan → confirm screen. Skip setup with pre-connect / pre-fill:

tales-tui --connect claude --connect codex
tales-tui --connect claude --prefill "Design a rate limiter"
tales-tui "Design and build a rate limiter" --drafter claude --critic codex
Roster rule. Connect Claude and Codex as many times as you like, but Open Code only once per run — its session continuity is keyed on the working directory, so two would resume each other's session. Tales rejects this roster up front.

04Scriptable commands

The non-interactive counterparts — for scripts, CI, and dogfooding.

tales solo — one agent, one turn

tales solo --model sonnet "Reply with exactly: hello from tales"
tales solo --agent codex --image diagram.png "Explain this architecture"

--agent claude|codex|opencode · --model · --cwd · --sandbox (Codex). --image (repeatable) attaches files: Claude receives images and PDFs; Codex receives images only; Open Code sees them named in the prompt text.

tales discuss — a live drafter/critic discussion, no execution

tales discuss --drafter claude --critic codex --turns 2 \
  --drafter-model sonnet --sandbox read-only \
  "Design a minimal rate limiter for a public REST API. 5 bullet points."

tales run — the full pipeline: discuss → recommend → execute

tales run "Build X" --drafter codex --critic claude --execute claude --turns 2

The recommendation is computed and printed, but --execute decides who runs (default claude). It can be one of the planners, or a different tool entirely. That choice is queued before the gate and auto-confirmed when the gate opens — the non-interactive equivalent of /confirm. There is no "execute the recommended pick" mode in a single invocation; --execute always decides.

Tiered execution — smart models plan, a cheap model runs

When --execute names a tool that isn't a planner, Tales enrolls it as a separate executor that sits out the debate and implements the agreed plan with its own model via --execute-model. Spend on judgment, not on typing:

# two strong models plan + argue, a cheap/fast one implements
tales run "add OAuth login" \
  --drafter claude --drafter-model opus \
  --critic  codex  --critic-model  gpt-5 \
  --execute opencode --execute-model <cheap-fast-model>

--execute-model is only valid for a separate executor; pointing --execute at one of the planners (same-tool tiering) is rejected for now. The executor sits out the discussion and the vote, but it's still a valid recommendation candidate.

Add --worktree to run that one executor in its own git worktree and merge the result back into your current branch (requires a git repo; otherwise ignored with a note):

tales run "Add a /health endpoint" --execute claude --worktree

In headless mode the Claude executor is restricted to file-writing tools (--allowedTools) so it can't stall on an unapproved shell prompt; Codex runs under its own sandbox policy instead.

This project's own landing/ page and these docs were built with Tales: Codex and Claude Code drafted and reviewed, via tales discuss.

tales open & the browser view

tales open opens the workspace in a new macOS window (used by the /tales skill from inside a harness); --connect <tool> pre-connects and an optional task pre-fills the prompt. tales-web is a secondary local axum + WebSocket view that streams the discussion and exposes an approve/reject gate — it currently supports Claude and Codex only:

tales-web "Design and build a rate limiter"   # → http://127.0.0.1:7878

05Architecture

A Cargo workspace with a strictly UI-agnostic core. Frontends exchange runtime events and commands with tales-core only through the bus — which is why the web frontend dropped in with zero changes to the engine.

AgentAdapter — the uniform interface over every tool

Each CLI implements one trait: spawn() launches the process and pumps normalized AgentEvents (TokenDelta, MessageFinal, ToolCall, TurnComplete, Error, …) into a channel, and returns a command sender driven with AgentCommands (StartTurn, InjectMessage, Shutdown). The engine and frontends see only these enums — never the concrete tool.

The tools differ sharply: Claude is bidirectional stream-json (read tokens live, inject mid-turn); Codex is turn-based exec + resume. That asymmetry lives only in AgentCaps flags (midturn_injection, structured_output, resumable, mcp) — the orchestrator schedules around capabilities, never if claude { … }. Adding a tool = one AgentAdapter impl + one row in KNOWN_TOOLS; the picker, CLI, and orchestrator all read that registry.

The bus — the one core↔frontend contract

EventBus owns a lossy broadcast channel for events (a slow frontend lags and recovers; it never blocks the engine) and an mpsc channel for commands (many interchangeable producers, one engine consumer). Clone it, hand a clone to each frontend, subscribe.

Worktree isolation

WorktreeManager can give each agent its own git worktree on its own branch; the orchestrator surfaces each worktree's diff (git diff --stat + full patch). Today tales run --worktree uses this for the single selected executor: create one worktree, run the executor there, and on success merge that branch back — Clean, NoChanges, or Conflict { files } (which leaves the base mid-merge for you to resolve). The executor's worktree is pruned on every exit path.

Recommendation & gate

After discussion, each agent emits a JSON recommendation — recommended_executor + confidence + rationale (tolerantly parsed from free-form model text). aggregate tallies confidence-weighted scores per candidate into a Recommendation, with a confident flag that's false when no real consensus emerged (so the UI can say "no clear pick" rather than fake one). The recommendation is advisory; execution begins only when you confirm interactively or via the queued --execute choice.

06Troubleshooting

  • Tool not connectable / "not found". The CLI isn't on PATH. Install it, reopen; Tales re-checks PATH per run.
  • Auth failures. Tales doesn't manage credentials — authenticate the underlying CLI directly (claude, codex, opencode) once, then retry.
  • "can't connect Open Code twice." Expected — see the roster rule. Use it as a single participant.
  • Only one tool installed. The default discussion needs two participants; pass --connect <tool> (TUI) or explicit --drafter/--critic to run with a specific roster.
  • --worktree ignored. The cwd isn't a git repo; Tales prints a note and runs without isolation. Run inside a repo to enable it.
  • Merge left mid-merge (Conflict). The executor's branch conflicts with your base; resolve and commit, or git merge --abort. A detached HEAD or dirty base can also block the merge.
  • Headless Claude executor seems limited. By design it's restricted to file-writing tools; a plan needing shell/Bash should run interactively in a live pane where you can approve.
